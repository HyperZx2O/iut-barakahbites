IUT Barakah Bites - System Architecture Documentation
===================================================

Architecture Paradigm:
---------------------
Distributed Microservices using Event-Driven and REST patterns.

Component Breakdown:
-------------------

1. Frontend (Vercel)
   - Student UI: Ordering interface for students.
   - Admin Dashboard: Monitoring and chaos management tool.

2. Order Gateway (Railway)
   - The primary entry point for order requests.
   - Flow: Validates JWT -> Checks Redis Stock Cache -> Calls Stock Service.
   - On success: Dispatches to Kitchen Queue AND return 202 Accepted.

3. Identity Provider (Railway)
   - Handles Registration, Login, and JWT issuance.
   - Implements Rate Limiting (Redis-backed window).
   - Manages Student Data in PostgreSQL.

4. Stock Service (Railway)
   - Source of truth for inventory.
   - Implements Optimistic Locking (versioning) to prevent race conditions.
   - Persistent storage in PostgreSQL.

5. Kitchen Queue (Railway)
   - Job queue (BullMQ/Redis) for decoupled processing.
   - Simulates preparation delay (3-7s).
   - Communicates order completion to Notification Hub.

6. Notification Hub (Railway)
   - Manages active SSE (Server-Sent Events) streams.
   - Broadcasts status updates globally using Redis Pub/Sub.

System Logic Flow Diagram (Text):
--------------------------------

[Student] --(Login)--> [Identity Provider]
[Student] --(Place Order + JWT)--> [Order Gateway]
                                       |
                   (Cache Check) <-- [Redis]
                                       |
                   (Transactional Dec) --> [Stock Service]
                                              |
                                     [PostgreSQL (Stock)]
                                       |
              (Publish Job) --> [Kitchen Queue]
                                       |
              (Status Update) --> [Notification Hub]
                                       |
              (SSE Stream) <--- [Student Tracking UI]

Data Consistency Patterns:
-------------------------
- Optimistic Locking: Stock service uses 'version' columns to handle concurrent requests.
- Idempotency: All ordering paths use 'orderId' as an idempotency key to prevent double charging/decrementing.
- Fail-Open Cache: Order Gateway treats Redis as a performance layer; if Redis is down, it bypasses to the Stock Service directly.

Deployment Topology:
-------------------
- Vercel: Global Edge Network for UI.
- Railway: Dedicated containers with private networking for backend services.
- Managed Plugins: Railway provides highly available PostgreSQL and Redis.
