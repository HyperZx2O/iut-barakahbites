## High‑Level Architecture Overview

The repository implements a **micro‑service based cafeteria ordering system** composed of the following services (each with its own Dockerfile, `package.json`, and tests):

| Service | Port | Primary Responsibility |
|---|---|---|
| **identity‑provider** | 3001 | Student authentication, JWT issuance, rate‑limited login |
| **order‑gateway** | 3002 | Validate JWT, cache‑first stock check, forward orders |
| **stock‑service** | 3003 | Persistent inventory, optimistic‑locking decrement, idempotency |
| **kitchen‑queue** | 3004 | Async order acknowledgement, simulate kitchen preparation, publish events |
| **notification‑hub** | 3005 | Server‑Sent Events (or WebSocket) push of real‑time order status |
| **student‑ui** (frontend) | 5173 | React app for students to login, place orders, track status |
| **admin‑dashboard** (frontend) | 5174 | React admin panel showing health grid, live metrics, chaos toggle |

All services communicate **exclusively over Docker’s internal network** using their service names (e.g., `http://stock-service:3003`). Environment variables (JWT secret, DB URLs, Redis URL, etc.) are injected via `docker‑compose.yml`.

### Interaction Flow (simplified)

1. **Login** – Student UI → `identity‑provider` `/auth/login` → receives JWT.
2. **Place Order** – UI → `order‑gateway` `/order` with Bearer token.
   - Gateway validates JWT, checks Redis cache for stock.
   - If stock is available, forwards to `stock‑service` to decrement.
   - On success, forwards order to `kitchen‑queue`.
3. **Kitchen Processing** – `kitchen‑queue` acknowledges immediately (202) and processes asynchronously, then publishes status updates to `notification‑hub`.
4. **Real‑time Updates** – UI opens SSE connection to `notification‑hub` and receives status events without polling.
5. **Admin Dashboard** – Queries `/health` and `/metrics` of each service, can invoke `/admin/kill` and `/admin/revive` for chaos testing.

## Suggested Development Workflow

1. **Start the full stack** with `docker compose up` to test integration.
2. **Iterate on a single service** locally (use `npm run dev` and hit the running Docker‑compose services for dependencies).
3. **Run unit tests** frequently (`npm test`). For CI‑style verification run the matrix: `gh workflow run ci.yml` or simply `docker compose run --rm <service> npm test`.
4. **Add a new test** or modify behavior, then run the relevant service’s test suite before committing.
5. **Commit** only after all local tests pass; the CI pipeline will enforce the same on push.

## Miscellaneous

- **Environment variables** are defined in `docker-compose.yml`; keep them in sync with local `.env` files if you run services outside Docker.
- **Rate‑limiting** and **idempotency** are implemented via Redis keys (`ratelimit:<studentId>`, `idempotency:<key>`).
- **Chaos endpoints** (`/admin/kill`, `/admin/revive`) are always reachable, even when the service is in the killed state.
- **Metrics** expose `totalOrders`, `ordersPerMinute`, `failedRequests`, `averageLatencyMs`, `p99LatencyMs`.
